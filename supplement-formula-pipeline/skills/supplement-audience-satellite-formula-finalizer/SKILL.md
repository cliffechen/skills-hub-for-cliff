# Supplement Audience Satellite Formula Finalizer

## 概览

这个 skill 是保健品开发链路的最后一环。它不负责上游风险查验，也不负责大范围发散新方向，而是把已经选中的方向压缩成一份**以受众为根基、可交给实验室初审的中文最终配方简报**。

它要回答的问题是：

- 这款新配方到底切的是谁？
- 这些人在哪里、希望得到什么、最痛的点是什么？
- 原产品的核心受众和核心功能是什么？
- 哪个成分是主星？
- 哪 1-2 个卫星卖点值得围绕主星增加？
- 工厂实验室第一轮需要核对什么？

## 在技能链中的位置

这个 skill 默认位于以下链路之后：

1. `ingredients-breakdown-compliance-check`
   作用：做成分、商标、专利、合规边界判断。
2. `supplement-safe-rebuild`
   作用：在安全边界内做方向重建。
3. `formula-reconstruction` 或边界分析工作流
   作用：做关键词、市场需求、用户反馈和候选方向判断。
4. `supplement-audience-satellite-formula-finalizer`
   作用：把已选方向收成最终配方简报和实验室对接清单。

不要用这个 skill 替代上游风险工作。如果上游风险结论不完整，只能基于明确证据做保守收口，并把缺口标成待确认项。

## 必要输入

尽量使用以下材料：

- 已选方向，例如 `方案 B`
- 参考 ASIN 或竞品
- 已确认的 Supplement Facts / 标签文字
- SIF 关键词解析结果
- Sorftime 的产品详情、流量词、关键词详情和评论
- 上游风险报告与 safe rebuild 简报
- solution comparison / formula reconstruction 简报
- 目标剂型、规格、市场和渠道

如果评论或受众证据缺失，应优先补齐再做最终收口。除非用户明确只要”纯配方草案”，否则**受众研究不是可选项**。

## 受众证据门槛条件（Gate Conditions）

在进入配方收口之前，必须满足以下最低证据要求。如果任一条件不满足，应优先补齐证据或明确告知用户当前缺口。

### 必须满足（全部满足才能进入收口）
1. **评论样本** — 至少 10 条评论（正面 ≥ 5，负面 ≥ 3），来源为 Sorftime product_reviews 或手动提供的评论文本
2. **关键词聚类** — 至少 3 个有效关键词聚类（来自 Sif Excel 解析或亚马逊搜索框下拉词分析）
3. **目标人群画像** — 至少明确一个核心人群定义（谁、在哪里触发、最痛的点是什么）
4. **对标产品信息** — 至少一个对标 ASIN 的产品详情和 Supplement Facts

### 可选但建议补充
- Reddit/社群讨论样本（≥ 5 条相关帖子）
- 竞品流量词交叉对比
- 上游 Product Research 报告

### 不满足时的处理
- 缺少评论 → 用 agent 内置 web search 搜索 `site:amazon.com {ASIN} reviews` 或请求用户提供截图
- 缺少关键词聚类 → 运行 `scripts/analyze_autocomplete.py` 或用 web search 搜索亚马逊下拉词
- 缺少人群画像 → 从评论和关键词反向推导，但标注为”推导画像，置信度：中”
- 用户明确要求跳过 → 标注”受众证据不完整，本次收口为保守版本，建议后续补齐后复核”

## 核心方法

### 1. 先做人群，再谈成分

在讨论任何新成分前，先明确：

- 原产品到底服务谁
- 这些人在哪里被触发：Amazon 搜索、评论、竞品词、社群、品类使用场景
- 他们希望得到什么
- 他们抱怨什么
- 他们害怕什么
- 他们信任原产品的什么点

新配方必须先证明自己能承接原产品受众，才有资格增加卫星卖点。

### 2. 定义主星

主星是承接原产品流量记忆和用户认知的核心成分或核心功能身份。

判断规则：

- 主星必须在视觉和概念上都占主位。
- 卫星成分不能反客为主，变成真正的产品身份。
- 主星要继续挂靠最强的关键词池。
- 如果原产品的剂量、剂型、递送方式或包装方式本身就是用户信任点，也要尽量保留。

### 3. 卫星卖点只加 1-2 个

卫星卖点必须围绕主星旋转，不能偏航。

好的卫星卖点：

- 真正解决受众痛点
- 让主成分更容易被理解
- 能打开相关而不是漂移的关键词承接
- 能改善耐受性、清洁标签、稳定性或使用体验
- 不突破 safe rebuild 的边界

不好的卫星卖点：

- 只是为了追高流量词
- 引入疾病、药物或治疗语义
- 带来明显相互作用或安全复杂度
- 把产品拉去另一个品类
- 需要很大剂量空间或制造复杂度，但收益不明确

### 4. 用专业膳食补充剂营养师视角做收口

收口时必须做简化版营养师审视：

- 角色是否清晰：每个成分到底负责什么
- 剂量是否讲得通：标签量是否合理、是否适合当前剂型
- 形式是否合理：原料形式是否与目标功能相匹配
- 是否冗余：有没有装饰性成分、重复成分
- 耐受性与安全边界：是否存在敏感人群、交互、上限或特别警示
- 声明是否克制：是否仍处在结构/功能语言范围内

这是产品开发评审，不是医疗建议，也不是最终生产批准。

### 5. 在功效、流量、简洁度之间求平衡

关键词数据用来帮助定位，不用来强迫我们加高风险成分。

优先级：

- 核心关键词承接
- 相邻关键词相关性
- 评论中的受众语言
- 配方简洁度
- 合规和 Amazon 可审核性

### 6. 最后一定要落到实验室清单

最终输出必须带一个简洁实验室对接清单，至少包括：

- 活性成分、建议形式、目标标签量
- 目标 serving size 和剂型
- 辅料目标和避免项
- 稳定性或释放要求
- 需要的文件：CoA、规格、过敏原、重金属、微生物、稳定性、释放测试
- 实验室要先回答的开放问题

清单要足够清楚，让实验室或配方工程师一眼知道先核什么。不要写成 clone-ready 制造工艺。

## 输出要求

正式交付物必须是中文。只有以下内容可以保留英文：

- 成分正式名称
- ASIN
- 核心关键词字面量
- 法规术语
- URL

必须包含以下章节：

1. `执行摘要`
2. `受众根基：人群、场景、痛点`
3. `原产品核心承接`
4. `主星-卫星架构`
5. `精简配方建议`
6. `营养师视角功能审视`
7. `关键词与人群承接`
8. `声明边界`
9. `实验室对接清单`
10. `剩余风险与待确认问题`

详细结构见 `references/output-template.md`。

## HTML 标签交付

当这个 skill 生成最终配方简报时，默认要伴随生成一份 HTML 格式的 Supplement Facts 标签，输出到同一个 `deliverables/final/` 目录。

默认文件名：

- `{name}-final-formula-brief.md`
- `{name}-final-supplement-facts.html`

这份 HTML 必须保持极简，只保留：

- `Supplement Facts` 主体
- `Other Ingredients`
- `Suggested Use`
- `Caution`

不要在 HTML 中加入以下内容，除非用户明确要求：

- 实验室对接说明
- 草案声明
- Free From 列表
- Lab Review Focus
- 额外营销文案
- 人群分析或配方解释

## 输出位置

这个 skill 从现在起默认只写入当前案例目录，不新建 `result/` 或其他根级目录。

唯一正式输出位置：

- `outputs/cases/{case}/deliverables/final/{name}-final-formula-brief.md`
- `outputs/cases/{case}/deliverables/final/{name}-final-supplement-facts.html`

默认生成这 2 份成对交付物：一份最终配方简报，一份极简 Supplement Facts HTML。

详细规则见 `references/output-contract.md` 和仓库根目录的 `docs/output-organization.md`。

## 护栏

- 不能从成分出发倒推人群，必须先看人群。
- 非用户明确要求时，不要加超过 1-2 个卫星卖点。
- 不要用疾病、治疗、替代药物的表述。
- 不要因为流量大就默认值得加成分。
- 不要覆盖上游 `avoid` 或 `legal review` 结论。
- 不要把配方写成“已可直接生产”。
- 不要提供详细制造参数、供应商复制路径或 clone-ready BOM。

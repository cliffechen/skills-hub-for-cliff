# 受众优先框架

当任务要求我们在最终定配方前先搞清楚“这款保健品到底是给谁”的时候，使用这个框架。

## 证据优先级

优先使用：

1. 参考 ASIN 及近邻竞品的 Amazon 评论
2. SIF 关键词相关性和高搜索低相关陷阱
3. Sorftime 的流量词和关键词详情
4. 标题、Supplement Facts、包装卖点和标签图
5. 只有在受众仍不清楚时，才补充站外讨论

## 受众抽取

先建立一个简洁的受众地图：

| 字段 | 要回答的问题 |
| --- | --- |
| 人群 | 谁在买，或者谁最可能买？ |
| 场景 | 他们在什么生活情境下会想到这个产品？ |
| 原因 | 他们为什么选原产品？ |
| 痛点 | 他们抱怨什么、害怕什么、觉得哪里不够？ |
| 信任点 | 原产品哪些资产被用户认可？ |
| 可承接性 | 新配方如何不丢掉这群人？ |

## 痛点分类

把痛点分成以下五类，避免混成一团：

- 功效痛点：无效、见效慢、效果衰减、功能闭环不完整
- 体验痛点：胃部不适、吞咽困难、味道、刺激感、服用不便
- 信任痛点：批次稳定性、包装、新鲜度、第三方检测、标签透明度
- 价格痛点：价格上涨、性价比、规格不合适
- 合规/安全痛点：药物交互、敏感人群、疾病暗示、上限或超量风险

## 主星 / 卫星测试

一个卫星卖点只有同时通过以下四关，才值得进入最终方案：

1. 受众测试：能解决真实受众痛点
2. 流量测试：能打开相关相邻流量，或提高核心词转化
3. 营养测试：有明确功能角色，且标签量讲得通
4. 风险测试：不会显著增加安全、声明、IP 或 Amazon 风险

任何一关没过，就降级为：

- `观察`
- `后续 SKU`
- `避免`

不要为了“看起来更强”就把没过关的成分硬塞进去。
